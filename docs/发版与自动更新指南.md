# 发版与自动更新指南（正式版 + 预发布双通道）

> 更新时间：2026-03-07  
> 适用背景：主干长期稳定、合并面板在候选分支持续验证

---

## 零、发布治理基线（先看这个）

### 0.1 通道规则（硬约束）

| 通道 | 允许分支 | GitHub Release 类型 | 更新源 |
|------|----------|---------------------|--------|
| 正式版 | `master` | 正式 Release | **正式 Gist** |
| 预发布 | 非 `master` 候选分支 | Pre-release | **测试 Gist** |

**硬规则**：

1. 非 `master` 分支禁止发正式版。
2. 预发布不得写入正式 `version.json`（避免普通用户收到测试更新）。
3. 上线前必须完成一次回滚演练并留痕。

### 0.2 分支建议

- `master`：正式主线  
- `release/v1-stable`：旧版稳定线（长期保留，仅收紧急修复）  
- `feature/merged-panel`：合并面板候选线（示例名）

## 二、发版流程（一键自动）

### 方式 A：正式版发布（`master`）

```bash
# 确认在主干
git checkout master
git pull origin master

# 补丁版本 1.0.2 → 1.0.3
python tools/release.py patch

# 次版本 1.0.2 → 1.1.0
python tools/release.py minor

# 主版本 1.0.2 → 2.0.0
python tools/release.py major

# 带更新日志
python tools/release.py patch -m "- 修复水面线计算bug\n- 新增批量导出功能"
```

### 方式 B：候选分支预发布（改造后命令）

> 以下命令依赖 `tools/release.py` 增加参数（见本文第九章改造方案）。

```bash
# 示例：在候选分支发 beta 预发布，不修改 version.py
git checkout feature/merged-panel
git pull origin feature/merged-panel

python tools/release.py patch \
  --prerelease \
  --no-bump \
  --gist-target test \
  --tag-suffix beta.20260307.1 \
  -m "- 合并面板预发布验证"
```

### 一键流程说明

正式版默认 6 步：
1. 运行空间轴线严格数学精度验收（失败即中止）
2. 递增 `version.py` 版本号
3. 打包（全量 + 补丁）
4. Git commit + tag + push
5. 创建 GitHub Release 并上传资产
6. 更新 Gist `version.json`

预发布（建议）：
1. 可运行精度验收（建议保留）
2. 默认 `--no-bump`（不消耗正式版本号）
3. 打包（可只发全量，或同样产出补丁）
4. Git commit/tag（带 beta 标识）
5. 创建 GitHub **Pre-release**
6. 更新**测试 Gist** `version.json`

## 四、手动分步（网络或自动化异常时）

```bash
# 1. 递增版本号 + 打包
python tools/build.py --bump patch

# 2. 手动 git commit + tag + push（当前仓库主干为 master）
git add version.py
git commit -m "release: v1.0.3"
git tag v1.0.3
git push origin master && git push origin v1.0.3

# 3. 手动到 GitHub 创建 Release，上传 dist/ 下的 zip
# 4. 手动编辑对应通道的 Gist version.json（正式/测试）
```

## 六、首次配置（仅需一次）

### 6.1 GitHub Token

1. 打开 https://github.com/settings/tokens/new  
2. 勾选权限：`repo` + `gist`  
3. 在项目根目录创建 `.env`：

```
GITHUB_TOKEN=ghp_xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx
```

### 6.2 双 Gist 配置

- 正式通道 Gist：`GIST_ID_PROD`
- 预发布通道 Gist：`GIST_ID_TEST`

> 预发布必须写 `GIST_ID_TEST`，不得写 `GIST_ID_PROD`。

## 七、用户端体验与安全

### 7.1 自动检查

- 程序启动后后台静默检查
- 有新版本时右上角 InfoBar 提示
- 状态栏显示更新可用信息

### 7.2 手动检查

- 点击侧边栏 `检查更新`
- 弹窗内提供“更新通道”下拉：`正式（稳定）` / `测试（预发布）`
- 选择测试通道时显示风险提示：测试版可能不稳定
- 正式通道支持“降级安装”：当本地高于正式版时，可手动回退到正式版
- 点击 `下载并更新`，升级场景补丁优先，失败回退全量

### 7.3 安全机制

- 补丁模式：覆盖变更文件，不删除无关文件
- 全量模式：先备份到 `_backup/`
- 授权文件 `.lic` 永不覆盖

### 7.4 每周同步防漂移（建议）

- 使用 `tools/git_publish_gui.py` 的“每周同步 master → 选中分支”按钮执行同步。
- 同步流程：`fetch` → `checkout` → `merge origin/master`（可选自动 push）。
- 工具会记录最近同步时间并显示状态；超过 7 天会给出提醒。
- 该流程用于降低候选分支长期漂移与遗忘风险。

## 八、回滚 SOP（生产必备）

### 8.1 发布前基线封存

```bash
git checkout master
git pull origin master
git tag -a vX.Y-pre-merge-YYYYMMDD -m "baseline before merged panel"
git push origin vX.Y-pre-merge-YYYYMMDD
```

### 8.2 三种回滚路径

**A. PR 未合并**：关闭 PR，停止推进。  
**B. 已合并**：`git revert <merge_or_squash_sha>` 回滚。  
**C. 严重异常**：从基线 tag 拉热修分支恢复旧版。

```bash
# 路径 B
git checkout master
git pull origin master
git revert <merge_or_squash_sha>
git push origin master

# 路径 C
git checkout -b hotfix/rollback-vX.Y vX.Y-pre-merge-YYYYMMDD
git push -u origin hotfix/rollback-vX.Y
```

### 8.3 回滚演练门禁

正式上线前必须验证：

1. 基线 tag 可恢复构建
2. `.qxproj` 新版文件可被旧版基本打开
3. `revert` 后可正常走正式发布流程

## 九、`tools/release.py` 参数改造方案（落地版）

### 9.1 目标

让同一脚本同时支持：

- 正式发布（`master` + 正式 Gist）
- 分支预发布（`prerelease` + 测试 Gist）

### 9.2 新增参数（建议）

| 参数 | 类型 | 默认 | 作用 |
|------|------|------|------|
| `--prerelease` | flag | false | 创建 GitHub pre-release |
| `--gist-target` | enum(`prod`,`test`) | `prod` | 选择更新正式或测试 Gist |
| `--no-bump` | flag | false | 不修改 `version.py` |
| `--tag-suffix` | str | 空 | 追加 tag 后缀（如 `beta.20260307.1`） |

### 9.3 行为约束（必须实现）

1. 非 `master` 分支且未指定 `--prerelease`：**直接报错退出**。
2. 指定 `--prerelease` 且 `--gist-target=prod`：**报错退出**。
3. 正式发布默认仍沿用当前行为（向后兼容）。

### 9.4 最小实现步骤

1. CLI 参数解析扩展。
2. 在创建 Release payload 时写入 `prerelease: true/false`。
3. Gist 更新逻辑按 `--gist-target` 选目标 ID。
4. 分支名检查与保护性阻断。
5. `--no-bump` 分支下复用当前版本号并跳过 bump。
6. `--tag-suffix` 参与 tag/name 生成（如 `v1.0.8.3-beta.20260307.1`）。

### 9.5 验收用例

| 用例 | 预期 |
|------|------|
| `master` 正式发布 | 成功，写正式 Gist |
| 非 `master` 无 `--prerelease` | 阻断并提示 |
| 非 `master` + `--prerelease --gist-target test` | 成功，写测试 Gist |
| `--prerelease --gist-target prod` | 阻断并提示 |
| `--no-bump` | `version.py` 不变 |

## 十一、更新源优先级

| 优先级 | 场景 | 更新源 |
|--------|------|--------|
| ① | 正式用户 | 正式 Gist + 正式 Release |
| ② | 测试用户 | 测试 Gist + Pre-release |

> 默认建议使用正式通道；用户可在“检查更新”弹窗手动切换测试通道。更新源不可用时应静默跳过，不影响程序正常使用。
